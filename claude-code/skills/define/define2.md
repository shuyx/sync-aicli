# Diagnose: 问题诊断和解决方案助手

## 概述

这个skill专门处理**问题诊断和解决方案**。当你遇到技术问题、系统故障、代码错误等问题时，这个skill会帮助你：

- 分析问题的根本原因
- 提供系统化的解决方案
- 自动记录诊断过程和解决方案
- 建立一个个人的问题解决知识库

## 支持的问题类型

该skill可以处理以下类型的问题：

- **错误信息问题** - 代码报错、系统错误、异常信息诊断
- **功能故障** - 功能不工作、性能下降、行为异常
- **配置问题** - 配置错误、环境问题、兼容性问题
- **集成问题** - 模块/系统集成失败、通信问题
- **性能问题** - 速度慢、内存溢出、资源泄漏
- **跨系统问题** - 多个组件交互导致的问题
- **最佳实践问题** - 如何优化、避免常见坑

## 处理流程

### 1. 问题识别

skill通过以下方式识别问题：

- **问题标记**：错误消息、bug、crash、not working等关键词
- **问题提取**：错误堆栈、日志信息、重现步骤
- **上下文理解**：涉及的系统、版本、环境信息

### 2. 诊断分析

对每个问题进行系统化分析，包括：

**在对话中：**
- **问题陈述** - 清晰复述理解的问题
- **根本原因分析** - 深入分析问题的根源（5个为什么法）
- **可能的解决方案** - 列出多个解决方案（从最可能到备选）
- **逐步实施步骤** - 详细的操作步骤
- **验证方法** - 如何确认问题已解决
- **预防措施** - 如何避免问题再次发生
- **相关资源** - 参考文档、代码示例等
- **长度**：无限制，根据问题复杂度尽可能详细

**格式指南：**
- 使用markdown清晰展示
- 包含代码示例和配置示例
- 使用表格对比不同解决方案
- 提供具体的命令或操作步骤

### 3. 文档保存

自动创建/更新markdown文件：

**文件组织：**
```
~/Obsidian/Obsidian Work/问题库/
├── [错误类型]/
│   ├── [具体问题].md
│   └── ...
├── [系统或工具名]/
│   ├── [问题描述].md
│   └── ...
└── ...
```

**文件命名规则：**
- 按错误类型或系统创建子目录
- 文件名：`[简洁问题描述].md`
- 例如：`Python/ModuleNotFoundError-numpy导入失败.md`

### 4. 响应格式

回答后提供：

1. **诊断总结** - 问题原因和推荐方案
2. **实施确认** - "已诊断并保存到 [filepath]"
3. **后续支持** - 提供验证和跟进的建议

## 核心原则

1. **系统分析** - 不只给答案，要分析根本原因
2. **多方案思维** - 提供多个解决方案，标明优缺点
3. **可操作性** - 提供清晰的步骤和代码示例
4. **知识积累** - 自动保存以供将来参考
5. **持续改进** - 记录解决过程和验证结果

## 配置

### 存储位置
默认存储位置：
- **主目录**: `~/Obsidian/Obsidian Work/问题库/`
- **备选**: 当前项目目录
- **可自定义**: 可通过Factory设置修改

### 语言支持
- 支持中文问题诊断
- Support for English problems
- 支持混合语言描述

### 分类方式

问题可按以下方式组织：
- 按**错误类型** - Python, Java, SQL, Network等
- 按**系统/工具** - Docker, Kubernetes, Git等
- 按**问题领域** - 性能优化, 安全问题, 集成问题等
- 按**时间序列** - 按诊断日期排序

## 使用示例

### 例1：代码错误

**你的问题**：
```
我的Python代码报错：
ModuleNotFoundError: No module named 'numpy'
我已经用pip install numpy了，为什么还是找不到？
```

**诊断流程：**
1. 分析：可能是多个Python环境、虚拟环境激活问题等
2. 提供5个解决方案，从最可能到备选
3. 给出详细的诊断步骤
4. 保存到：`问题库/Python/ModuleNotFoundError-numpy导入失败.md`

### 例2：系统集成问题

**你的问题**：
```
Docker容器无法连接到宿主机的数据库
连接超时，网络看起来正常
```

**诊断流程：**
1. 分析：可能是网络配置、防火墙、DNS等问题
2. 列出检查清单和诊断命令
3. 提供不同场景的解决方案
4. 保存到：`问题库/Docker/容器数据库连接超时.md`

### 例3：性能问题

**你的问题**：
```
我的应用启动很慢，怎么诊断和优化？
```

**诊断流程：**
1. 分析：需要profiling来找瓶颈
2. 提供profiling工具和方法
3. 列出常见的性能瓶颈和优化方案
4. 保存到：`问题库/性能优化/应用启动速度优化.md`

## 最佳实践

### 问题描述建议
1. **提供错误信息** - 完整的错误堆栈或日志
2. **描述环境** - 系统版本、软件版本等
3. **说明重现步骤** - 其他人可以复现
4. **说明已尝试** - 你已经尝试过什么解决方案

### 诊断验证
1. 按照给定的步骤操作
2. 使用验证方法确认问题已解决
3. 记录最终的解决方案和耗时

### 知识库维护
1. 定期复习已保存的问题
2. 更新新发现的解决方案
3. 标记已验证可行的方案
4. 建立问题之间的关联

## 与其他Skills的集成

- **define** - 先定义相关概念，再诊断问题
- **content-research-writer** - 将诊断结果扩展为文档
- **docx** - 导出诊断报告为专业文档
- **canvas-design** - 创建问题诊断流程图或决策树

## 局限性

- 专注于**问题诊断和解决**，不包含实现指导
- 基于**描述进行诊断**，复杂问题可能需要更多上下文
- 保存到**本地markdown**，不支持云端同步
- 需要**清晰的问题描述**以进行准确诊断
