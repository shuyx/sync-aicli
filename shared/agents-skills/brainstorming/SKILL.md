# Brainstorming — 对话式想法到方案

将模糊想法通过结构化对话，转变为清晰的设计规格或商业方案。

## 核心原则

- **一次只问一个问题** — 绝不同时问多个，给对方思考空间
- **选择题优先** — 给出 2-3 个选项比开放式问题更高效
- **YAGNI 原则** — 无情删掉"以后可能用到"的功能，只保留当前需要的
- **先探索再设计** — 理解清楚问题之前，不给出方案
- **分段验证** — 每 200-300 字停下确认，而不是一次给出全部

---

## 工作流

### 阶段 1：理解问题（Understanding Phase）

首先读取当前上下文（代码库 / 文档 / 近期对话记录），然后**一次一个问题**探索：

```
探索维度（按顺序，每次只问一个）：
1. 目的：这是为了解决什么问题？
2. 用户：谁会用它？
3. 约束：有什么限制（时间/技术/资源）？
4. 成功标准：什么算做好了？
5. 对比：现在是怎么做的？为什么不够？
```

**示例开场**：
> 「听起来很有意思！先帮我理解一下—— 这个功能主要是为了解决什么问题？」

---

### 阶段 2：探索方案（Approaches Phase）

理解问题后，提出 **2-3 个可选方案**，带权衡分析：

```markdown
**方案 A（推荐）**：[名称]
- 做法：...
- 优点：...
- 代价：...

**方案 B**：[名称]
- 做法：...
- 优点：...
- 代价：...

我推荐方案 A，因为 [理由]。你觉得呢？
```

---

### 阶段 3：呈现设计（Design Phase）

确认方向后，**分段**呈现设计（每段 200-300 字，每段后询问确认）：

分段顺序：
1. 整体架构和核心流程
2. 关键组件和数据结构
3. 接口/API 设计
4. 错误处理和边界情况
5. 测试策略

每段结尾：「这部分看起来对吗？还是有需要调整的地方？」

---

### 阶段 4：存档（Documentation Phase）

方案确认后，自动存档：

```bash
# 技术方案存档路径
docs/plans/YYYY-MM-DD-<topic>-design.md

# 商业/产品方案存档路径（Obsidian）
~/Obsidian/kevinob/brainstorm/YYYY-MM-DD-<topic>.md
```

文档结构：
```markdown
# [主题] 设计方案
日期：YYYY-MM-DD

## 问题背景
## 方案选择（含权衡）
## 最终方案详情
## 待确认事项
## 下一步行动
```

## 场景适配

### 技术开发场景
- 读取当前代码库结构作为上下文
- 关注：架构 / 组件 / 数据流 / 错误处理 / 测试
- 结束后提供：实现计划 + git 工作区设置建议

### 产品/商业场景
- 关注：用户价值 / 商业逻辑 / 竞品对比 / 资源约束
- 可调用 `deep-search` skill 补充市场调研
- 结束后提供：PRD 草案 / 路线图建议

### 创意场景（写作/设计/营销）
- 关注：受众 / 调性 / 传播渠道 / 成功指标
- 可调用 `content-research-writer` skill 输出成品

---

## 反模式（避免）

❌ 一次问多个问题
❌ 没理解问题就给方案
❌ 让用户填空式开放问题
❌ 给出超长方案一次性输出不验证
❌ 加入"以后可能用到"的功能
