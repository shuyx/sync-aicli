# Knowledge Corpus Builder

> 批量阅读源文档 → 分类 → 提取知识点 → 生成体系化知识库文档（Obsidian Markdown）

## 核心路径

```
源文档文件夹 (Cubox/xxx/)
    ↓ scan → read → classify → extract → generate
输出文档文件夹 (学习提高/xxx/)
```

## 重要约定

- **Obsidian 根目录**: `~/Obsidian/kevinob/`
- **源文档**: 通常在 `Cubox/<topic>/` 或 `Cubox/<topic>/Knowledge/`
- **输出目录**: `学习提高/<topic>/`
- **状态文件**: `/tmp/kcb-state-<topic>.json`（断点续作用）

## Phase 1：扫描与分类

### 1.1 扫描源文档

1. 用 `find_by_name` 或 `list_dir` 列出所有 `.md` 文件
2. 检查是否有状态文件 `/tmp/kcb-state-<topic>.json`
   - 有 → 断点续作，跳过已处理的文件
   - 无 → 首次执行，创建状态文件

### 1.2 快速分类（先标题/首段扫描）

对于大规模文件夹（> 100 篇），不要逐篇全文阅读。先做标题/首段扫描：

1. 每次用 `view_file` 只读每篇文档的前 30 行（标题 + 摘要）
2. 每批处理 20-30 篇的标题/首段
3. 基于标题和首段，将文档分入候选技术方向
4. 建立初步分类 taxonomy 并记录到状态文件

### 1.3 建立分类 Taxonomy

生成一个技术方向分类表，例如：

```json
{
  "taxonomy": {
    "方向1-名称": {
      "description": "该方向的简述",
      "doc_count": 45,
      "docs": ["file1.md", "file2.md", ...]
    },
    "方向2-名称": { ... }
  }
}
```

**分类原则**：
- 方向数量控制在 **8-15 个**
- 每个方向至少有 **5 篇**相关文档才独立成方向
- 不足 5 篇的归入「其他/综合」方向
- 同一篇文档可归入多个方向（标注主方向和次方向）

## Phase 2：深读与提取

### 2.1 按方向批量深读

每个技术方向内，逐篇深读文档：

1. 用 `view_file` 阅读全文
2. 提取以下信息：
   - **核心概念/定义** — 术语、原理、公式
   - **技术方案/算法** — 实现方法、架构设计
   - **实践经验** — 参数选择、踩坑记录、对比评测
   - **商业/产品化价值** — 市场分析、应用场景、商业模式
   - **代码/工具** — 开源项目、框架、库
   - **[模式B/C] 项目关联** — 与锚定项目的联结点

3. 每读完一篇，将提取结果暂存到状态文件的对应方向下

### 2.2 每轮处理量

- **每轮会话**处理 **1-2 个技术方向**（约 20-50 篇文档的深读）
- 每轮结束时报告进度并更新状态文件
- 告知用户下次会话的续作指令

### 2.3 [模式C 特有] 优先级排序

Gap-Analysis 模式下，按以下优先级排序深读文章：
1. **直接解决已知问题**的文章 → 最高优先
2. **补强技术短板**的文章 → 高优先
3. **拓展新能力**的文章 → 中优先
4. **背景知识**的文章 → 低优先

## Phase 3：体系化生成

### 3.1 文档结构

每个技术方向生成一篇大文档，使用模板：
```
skills/knowledge-corpus-builder/templates/topic_document.md
```

### 3.2 总览索引

所有方向文档生成完毕后，生成总览索引，使用模板：
```
skills/knowledge-corpus-builder/templates/overview_index.md
```

### 3.3 输出文件结构

```
学习提高/<topic>/
├── 00-<topic>技术学习体系总览.md     ← 总览索引
├── 01-<方向1名称>.md
├── 02-<方向2名称>.md
├── ...
├── NN-<方向N名称>.md
└── 99-完成报告.md
```

### 3.4 知识溯源标注

每个知识段落末尾标注来源：

```markdown
### 某个知识点

具体内容段落...

> 📎 **来源**: [[Cubox/<topic>/Knowledge/文章A.md]] | [[Cubox/<topic>/文章B.md]]
```

### 3.5 [模式B] 项目联结标注

```markdown
### 某个知识点

具体内容段落...

> 🔗 **项目关联**: 可直接应用于 [[项目路径/模块A]]，解决 XXX 问题
> 📎 **来源**: [[Cubox/...]]
```

### 3.6 [模式C] 补缺方案标注

```markdown
### 某个知识点

具体内容段落...

> 🎯 **补缺价值**: 解决项目中 XXX 的技术缺口，优先级 ⭐⭐⭐⭐⭐
> 📎 **来源**: [[Cubox/...]]
```

### 3.7 交叉引用

在文档内使用 Obsidian `[[]]` 双链：
- **同 topic 内**: `参见 [[02-方向B#子章节名]]`
- **跨 topic**: `与 [[学习提高/另一个topic/01-xxx]] 中的 YYY 技术相关`

## Phase 4：收尾

### 4.1 生成完成报告

参考模板中的完成报告格式，包含：


## 断点续作

### 识别续作请求

当用户说类似以下内容时，识别为续作：
- "继续整理 XXX"
- "接着上次 XXX"
- "继续 knowledge corpus builder"

### 续作流程

1. 读取 `/tmp/kcb-state-<topic>.json`
2. 检查进度状态（scan/read/generate 哪个阶段）
3. 从上次中断处继续
4. 显示进度摘要：

```
📊 续作进度:
- 源文档: N 篇（已扫描 X，已深读 Y）
- 分类: M 个方向
- 已生成: K 篇文档
- 当前阶段: <阶段描述>
- 下一步: <接下来要做的事>
```

### 状态文件格式

```json
{
  "version": "1.0",
  "topic": "智能制造",
  "mode": "project-anchored",
  "project_path": "商业孵化/【10】矿山自动化/",
  "source_dir": "Cubox/智能制造/",
  "output_dir": "学习提高/智能制造/",
  "phase": "read",
  "total_docs": 509,
  "scanned_docs": 509,
  "classified_docs": 509,
  "read_docs": 85,
  "taxonomy": {
    "01-传感器融合": {
      "docs": ["doc1.md", "doc2.md"],
      "read": ["doc1.md"],
      "status": "in_progress"
    }
  },
  "generated_files": [
    "学习提高/智能制造/01-传感器融合.md"
```

## 增量更新

当用户对已完成的 topic 执行增量更新时：

1. 读取已有的状态文件
2. 扫描源文件夹，识别新增文章（比对已记录的文件列表）
3. 对新增文章执行 scan→classify→read→extract
4. 将新知识点追加到已有方向文档的对应章节
5. 更新总览索引
6. 记录变更日志

---

## 质量检查清单

生成文档后，自检以下项目：

- [ ] 每个技术方向的知识点是否覆盖了所有相关来源文章
- [ ] 溯源链 `📎 来源` 是否指向有效的 Obsidian 路径
- [ ] 交叉引用 `[[]]` 双链是否指向实际存在的文档
- [ ] [模式B/C] 项目关联标注是否有具体的联结描述
- [ ] 总览索引是否覆盖了所有方向文档
- [ ] 输出文档编号是否连续且无遗漏
